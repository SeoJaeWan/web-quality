Accessibility Issues (6)

A1 — Icon Button Missing Accessible Name High
Line 29-31 · Rule: aria-label-missing
The first icon button contains only an <svg> element with no text content or aria-label. Screen readers will announce this as an empty or unlabeled button, making it impossible for assistive technology users to understand the button's purpose.
Fix: Add aria-label describing the action, e.g. <button aria-label="Toggle grid view">. Mark the SVG with aria-hidden="true" since the label is on the button.
A2 — Second Icon Button Missing Accessible Name High
Line 34-36 · Rule: aria-label-missing
Same issue as A1. The second <button> wraps an SVG without any accessible name. This violates WCAG 2.1 SC 4.1.2 (Name, Role, Value) and SC 1.1.1 (Non-text Content).
Fix: Add aria-label to the button and aria-hidden="true" to the SVG.
A3 — SVG Missing Role or aria-hidden (First Icon) Medium
Line 30 · Rule: svg-missing-role
The SVG element lacks role="img" or aria-hidden="true". Without these, some screen readers may attempt to traverse the SVG's internal elements (like <circle>), producing confusing output.
Fix: For decorative SVGs, add aria-hidden="true". For meaningful SVGs, add role="img" and a <title> element inside the SVG.
A4 — SVG Missing Role or aria-hidden (Second Icon) Medium
Line 35 · Rule: svg-missing-role
Same issue as A3, affecting the second SVG icon.
Fix: Add aria-hidden="true" to the SVG element.
A5 — Navigation Landmark Missing Label Low
Line 24 · Rule: nav-missing-label
The <nav> element lacks an aria-label or aria-labelledby attribute. When a page has multiple navigation landmarks, users of assistive technology cannot distinguish between them.
Fix: Add aria-label="Main navigation" or use aria-labelledby pointing to the heading.
A6 — Image Missing Explicit Dimensions Low
Line 50 · Rule: img-dimensions-missing
The <img> tag has an alt attribute (good), but lacks explicit width and height. While primarily a performance concern, missing dimensions also affect assistive tech users who may experience disorienting layout shifts.
Fix: Add width and height attributes to the <img> element.

Performance Issues (7)

P1 — Layout Thrashing in useEffect High
Line 12-13 · Rule: layout-thrashing
The code reads offsetHeight (forcing the browser to calculate layout) and then immediately writes to style.minHeight (invalidating that layout). This is a classic forced synchronous reflow pattern. In a more complex component or in a loop, this can severely degrade frame rate and responsiveness.
Fix: Separate reads and writes. Use requestAnimationFrame to batch the write, or use ResizeObserver to react to size changes without forcing reflow. If the measurement must happen before paint, use useLayoutEffect instead.
P2 — Cumulative Layout Shift (CLS) from Delayed Content High
Line 17-19 · Rule: cls-risk
A setTimeout injects new list items after 500ms without any space reservation. When the items render, all content below shifts downward. This directly harms the CLS Core Web Vital metric and creates a jarring user experience.
Fix: Reserve space with a skeleton placeholder or min-height on the container. Better yet, load data before initial render (SSR/SSG) or show a loading state with fixed dimensions.
P3 — Image Missing loading="lazy" Medium
Line 50 · Rule: img-lazy-loading
The image at the bottom of the page (below the fold) is loaded eagerly by default. This wastes bandwidth on initial page load and can delay Largest Contentful Paint (LCP) for above-the-fold content by competing for network resources.
Fix: Add loading="lazy" to the image: <img loading="lazy" src="..." alt="..." />
P4 — Image Missing Width/Height (CLS Contributor) Medium
Line 50 · Rule: img-no-dimensions
Without explicit width and height, the browser cannot calculate the image's aspect ratio before it loads. Once loaded, the image expands and pushes content below it, contributing to CLS.
Fix: Add width and height attributes matching the image's intrinsic dimensions, or use CSS aspect-ratio.
P7 — useEffect Instead of useLayoutEffect for DOM Measurement Medium
Line 9-20 · Rule: useEffect-side-effects
useEffect runs after the browser paints. Since lines 12-13 measure and mutate the DOM, the user will see the initial render, then a flash/shift when the minHeight is applied. This creates a visible layout jump.
Fix: Use useLayoutEffect for DOM measurements that affect visual layout. It fires synchronously after DOM mutations but before the browser paints, preventing the flash.
P5 — Missing Key Prop on List Items Low
Line 45-47 · Rule: missing-key-prop
The .map() call renders <div> elements without a key prop. React will fall back to index-based reconciliation, which can cause unnecessary DOM mutations and re-renders when items change, reducing rendering performance.
Fix: Add a stable, unique key: items.map((item) => <div key={item} ...>). If items may not be unique, use a unique identifier from your data.
P6 — Image Lacks Responsive srcSet Low
Line 50 · Rule: img-no-srcset
The image uses a single src with no srcSet or <picture> element. On high-DPI displays, this may serve a blurry image; on mobile, it may serve an unnecessarily large image, wasting bandwidth.
Fix: Use srcSet and sizes attributes, or wrap in a <picture> element to serve appropriately sized images per viewport.
