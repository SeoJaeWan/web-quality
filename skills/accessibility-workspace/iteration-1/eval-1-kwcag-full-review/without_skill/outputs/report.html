# File Line Severity KWCAG 2.2 Guideline Issue Description Recommendation
1 index.html 30 Critical 1.1.1 Appropriate alternatives for non-text content (Non-text content alternatives) <img src="/public/logo.png"> is missing the alt attribute entirely. Screen readers cannot convey the meaning of this image to users. Add a descriptive alt attribute. If the image is decorative, use alt="" and role="presentation". E.g., <img src="/public/logo.png" alt="Company logo" width="120" height="40" />
2 dashboard.tsx 29-31 Critical 1.1.1 Appropriate alternatives for non-text content (Non-text content alternatives) Icon buttons contain only <svg> elements with no accessible name. The <button> elements have no aria-label, aria-labelledby, or visible text. Screen reader users cannot determine the button's purpose. Add aria-label to each button describing its action. Also add aria-hidden="true" to the SVG icons. E.g., <button aria-label="Settings">
3 Card.tsx 24-30 Critical 2.1.1 Keyboard accessibility (Keyboard usage guarantee) The <div> element has an onClick handler but is not keyboard accessible. It lacks tabIndex, role, and onKeyDown/onKeyPress handlers. Keyboard-only users cannot activate this interactive element. Either replace the <div> with a <button> or add role="button", tabIndex={0}, and an onKeyDown handler that triggers on Enter/Space. Using a semantic <button> is preferred.
4 Card.tsx 27 Critical 4.1.1 Parsing (Markup validity) id="card-item" is hardcoded on the Card component. When multiple Cards are rendered in a list, duplicate IDs are produced, violating HTML specification. This breaks aria-labelledby / aria-describedby references and assistive technology parsing. Generate unique IDs per instance using useId() (React 18+) or pass a unique ID via props. E.g., id={`card-item-${id}`}
5 index.html 2 Critical 3.1.1 Language of page (Primary language specification) <html lang="en"> is set but if the page content is in Korean, the language attribute does not match the actual content language. Screen readers will use incorrect pronunciation rules. Set the lang attribute to match the primary language of the page content. If the page is in Korean, use lang="ko". If mixed, use lang attributes on specific elements for language changes.
6 dashboard.tsx 30 Major 1.1.1 Appropriate alternatives for non-text content (Non-text content alternatives) SVG icons inside buttons lack <title> elements and aria-hidden="true". Without aria-hidden, screen readers may attempt to read SVG markup content, causing confusion. Add aria-hidden="true" to decorative SVGs, or add a <title> element inside the SVG for meaningful icons. The parent button's aria-label should provide the accessible name.
7 index.html 20-44 Major 1.3.1 Info and relationships (Content structure and semantics) The page body lacks semantic landmark elements. There is no <main>, <nav>, <header>, or <footer>. Content is placed directly inside <body> without structural organization. Assistive technology users cannot navigate by landmarks. Wrap the primary content in <main>. Add <header> for the heading area and <nav> for navigation links. This enables landmark-based navigation.
8 dashboard.tsx 41 Major 1.3.1 Info and relationships (Content structure and semantics) The <section> element lacks an accessible name. Sections used as landmarks should have an aria-label or aria-labelledby attribute to be meaningful to assistive technology. Add aria-labelledby="overview-heading" to the section and give the <h2> an id="overview-heading". Alternatively, add aria-label="Overview" to the section.
9 index.html 24, 35 Major 2.4.4 Link purpose (Link text clarity) Link texts "About Us" and "latest deals" are somewhat generic. While they have some context, <a href="...">latest deals</a> alone may not be sufficient out of context for screen reader users navigating by links list. Ensure link text is descriptive enough to understand in isolation. Consider adding aria-label for more context if needed, e.g., aria-label="View our latest deals and promotions".
10 Card.tsx 24-30 Major 4.1.2 Name, Role, Value (Component name, role, value) The clickable <div> in Card has no ARIA role. Interactive elements must expose their role to assistive technologies. Without role="button", the element is perceived as a static container. Add role="button" if using a <div>, or preferably refactor to use a native <button> element which inherently has the correct role, name, and keyboard behavior.
11 dashboard.tsx 45-46 Major 1.3.1 Info and relationships (Content structure and semantics) Dynamically rendered list items (items.map) are rendered as plain <div> elements inside a grid. If these represent a list of items, they should use a semantic list structure (<ul>/<li>) so screen readers announce the count and position. Wrap the mapped items in a <ul> and use <li> for each item, or use appropriate ARIA roles (role="list" and role="listitem").
12 index.html 37-44 Minor 2.1.1 Keyboard accessibility (Keyboard usage guarantee) The inline script uses innerHTML with user input (location.hash) which is an XSS risk. While primarily a security issue, injected content could disrupt the accessibility tree and break assistive technology parsing. Use textContent instead of innerHTML to prevent script injection. E.g., label.textContent = "Hello, " + userInput;
13 index.html 21 Minor 2.4.2 Page titled (Page title provision) The page title "Sample Page" is generic and non-descriptive. Page titles should clearly identify the page content and site to help users understand where they are, especially when switching between tabs. Use a descriptive, unique page title that identifies both the page content and the website. E.g., <title>Homepage - Company Name</title>
14 dashboard.tsx 16-19 Minor 3.2.1 On focus / 3.2.2 On input (Predictable behavior) Content is injected after a 500ms setTimeout without any notification to assistive technologies. Dynamically added content should be announced via ARIA live regions so screen reader users are aware of the change. Use an aria-live="polite" region to wrap the dynamically inserted content, or use aria-atomic="true" if the entire region should be re-announced. E.g., <section aria-live="polite">
15 Card.tsx 29 Minor 2.4.7 Focus visible (Visible focus indicator) The clickable Card <div> does not appear to have focus styling. Even if tabIndex is added, there is no :focus or :focus-visible CSS class applied. Users navigating by keyboard cannot see which element is focused. Add a visible focus indicator style. E.g., add a Tailwind class like focus:ring-2 focus:ring-blue-500 focus:outline-none or define :focus-visible CSS styles.
16 index.html 20-44 Info 2.4.1 Bypass blocks (Skip navigation) No skip navigation link is provided. If the page grows to include a navigation bar, users relying on keyboard or screen readers would need to tab through all navigation links to reach the main content. Add a visually hidden skip link at the top of the page: <a href="#main-content" class="sr-only focus:not-sr-only">Skip to main content</a> and assign id="main-content" to the main content area.
17 dashboard.tsx 24 Info 2.4.1 Bypass blocks (Skip navigation) The dashboard has a <nav> followed by a <section>. While landmarks help, a skip link would improve efficiency for keyboard users to jump directly to the main content area. Consider adding a skip navigation mechanism, particularly if the navigation grows in complexity. Landmark roles (<main>, <nav>) already provide some bypass capability for screen reader users.
