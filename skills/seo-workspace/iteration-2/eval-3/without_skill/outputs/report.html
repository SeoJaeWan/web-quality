CLS (Cumulative Layout Shift) Issues

HIGH Dynamic content insertion without size reservation CLS

dashboard.tsx : lines 16–19
setTimeout으로 500ms 후 items 상태를 업데이트하여 3개의 <div>를 삽입하지만, 해당 영역에 미리 확보된 공간이 없다. 콘텐츠가 삽입되면서 아래쪽 이미지와 기타 요소가 밀려나 CLS가 발생한다.
Recommendation: 삽입될 콘텐츠 영역에 min-height를 CSS로 미리 지정하거나, skeleton placeholder 컴포넌트를 렌더링하여 공간을 예약한다. 서버 사이드 렌더링(SSR)으로 초기 데이터를 포함시키는 것도 좋은 방법이다.

HIGH <img> width/height 미지정 CLS

dashboard.tsx : line 50
<img src="/public/promo.png" alt="Promo" className="mt-8" />에 width와 height 속성이 없다. 브라우저가 이미지 로드 전까지 해당 요소에 0px 높이를 할당하므로, 이미지가 로드되면 레이아웃이 급격히 변경(CLS)된다.
Recommendation: 명시적 width/height 속성을 추가한다. 반응형 레이아웃이라면 CSS aspect-ratio를 함께 사용한다. Next.js 프로젝트라면 next/image 컴포넌트를 사용하면 자동으로 처리된다.
<img src="/public/promo.png" alt="Promo" width={800} height={400} className="mt-8" />

Image Loading Optimization

HIGH Below-fold 이미지에 loading="lazy" 미적용 Image Optimization

dashboard.tsx : line 50
페이지 하단(mt-8)에 위치한 프로모 이미지에 loading="lazy"가 없어, 사용자가 스크롤하기 전에도 이미지가 즉시 다운로드된다. 이는 초기 로딩 시간(LCP)에 악영향을 미치고 불필요한 대역폭을 소비한다.
Recommendation: <img loading="lazy" decoding="async" .../>를 추가하여 뷰포트에 진입할 때만 이미지를 로드한다. 단, LCP 대상 이미지(above-the-fold)에는 loading="lazy"를 사용하지 않아야 한다.

MEDIUM srcSet/sizes 미사용 (반응형 이미지 미지원) Image Optimization

dashboard.tsx : line 50
단일 src만 사용하여 모든 디바이스에서 동일한 원본 이미지를 다운로드한다. 모바일 사용자가 데스크톱용 고해상도 이미지를 다운로드하게 되어 데이터 낭비와 로딩 지연이 발생한다.
Recommendation: srcSet과 sizes 속성을 추가하여 디바이스 해상도에 맞는 이미지를 제공한다.
<img srcSet="/promo-400.webp 400w, /promo-800.webp 800w" sizes="(max-width: 600px) 400px, 800px" .../>

MEDIUM 차세대 이미지 포맷(WebP/AVIF) 미사용 Image Optimization

dashboard.tsx : line 50
PNG 포맷만 사용하고 있다. WebP는 PNG 대비 평균 26% 더 작고, AVIF는 최대 50% 더 작은 파일 크기를 제공한다.
Recommendation: <picture> 요소를 사용하여 차세대 포맷을 우선 제공하고 PNG를 폴백으로 둔다.
<picture><source srcSet="/promo.avif" type="image/avif"/><source srcSet="/promo.webp" type="image/webp"/><img src="/promo.png" .../></picture>

Rendering Performance

HIGH Layout thrashing (강제 리플로우) Rendering Performance

dashboard.tsx : lines 10–14
useEffect 내에서 offsetHeight 읽기(강제 리플로우 트리거) 직후 style.minHeight를 설정(쓰기)하여 연속적인 강제 리플로우가 발생한다. 이 패턴은 "layout thrashing"으로 알려져 있으며, 복잡한 레이아웃에서는 심각한 성능 저하를 유발한다.
Recommendation: requestAnimationFrame을 사용하여 읽기/쓰기를 분리하거나, ResizeObserver를 활용한다. 가능하다면 CSS만으로 min-height를 선언적으로 지정하는 것이 가장 좋다.

MEDIUM 리스트 항목에 key prop 누락 Rendering Performance

dashboard.tsx : lines 45–47
items.map()에서 key prop이 없다. React가 리스트 변경 시 어떤 항목이 추가/제거/변경되었는지 효율적으로 판단하지 못해 전체 리스트를 다시 렌더링한다. 콘솔에 경고도 발생한다.
Recommendation: 고유한 key prop을 추가한다:
{items.map((item) => <div key={item} className="border p-2">{item}</div>)}

LOW document.querySelector로 직접 DOM 조작 Rendering Performance

Card.tsx : lines 17–20
React의 가상 DOM을 우회하여 document.querySelector로 직접 DOM 요소를 선택하고 스타일을 변경한다. 이는 React의 렌더링 최적화(reconciliation)를 방해하고 예기치 않은 리페인트를 유발할 수 있다. 또한 Card가 여러 개 렌더링되면 첫 번째 .card-highlight만 영향받는 버그도 있다.
Recommendation: useRef를 사용하여 특정 요소를 참조하고, useState로 상태를 관리하여 스타일을 선언적으로 적용한다.

LOW 프로덕션 코드에 console.log 잔존 Rendering Performance

Card.tsx : line 13
이벤트 핸들러에 console.log가 남아 있다. 빈번히 호출되는 핸들러에서는 직렬화 비용이 누적될 수 있으며, DevTools가 열려 있을 때 성능 영향이 더 크다.
Recommendation: 프로덕션 빌드에서 console.log를 제거한다. Babel 플러그인(babel-plugin-transform-remove-console) 또는 빌드 도구 설정으로 자동 제거할 수 있다.
