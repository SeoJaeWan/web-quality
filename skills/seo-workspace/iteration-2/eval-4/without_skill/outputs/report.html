1. JavaScript Loading Issues

Inline render-blocking script High
Lines 37-44
A synchronous inline <script> block in <body> blocks HTML parsing and rendering. The script executes immediately upon encounter, delaying First Contentful Paint (FCP) and Largest Contentful Paint (LCP). This directly harms the user's perceived loading speed.
Recommendation: Move the script to an external file and load it with the defer attribute: <script defer src="app.js"></script>. For non-critical scripts, consider async or dynamic import().
innerHTML usage (XSS and performance risk) Medium
Line 42
Using innerHTML triggers the HTML parser and can cause unnecessary reflows. Combined with unsanitized user input from location.hash, this creates a Cross-Site Scripting (XSS) vulnerability. An attacker could inject malicious HTML/JS via the URL hash.
Recommendation: Replace innerHTML with textContent to avoid XSS risk and reduce parsing overhead. Always sanitize user input before any DOM insertion.
No defer/async script loading strategy High
Line 37
The page lacks any external scripts loaded with defer or async. There is no script loading optimization strategy. All JavaScript runs synchronously and blocks the rendering pipeline.
Recommendation: Externalize scripts and use <script defer src="..."> to allow HTML parsing to continue while scripts download in parallel. Use async for independent third-party scripts.

2. Font Optimization Issues

Missing font-display descriptor High
Lines 13-17
The @font-face declaration for CustomFont does not include a font-display property. Without this, browsers default to font-display: auto, which typically causes a Flash of Invisible Text (FOIT) -- text is completely hidden until the custom font finishes downloading. This severely impacts FCP and LCP.
Recommendation: Add font-display: swap; to the @font-face rule. This tells the browser to immediately render text with a fallback system font, then swap to the custom font once loaded:
@font-face { font-family: "CustomFont"; src: url("/fonts/custom.woff2") format("woff2"); font-display: swap; }
No font preloading Medium
Not present in file
The custom font file (/fonts/custom.woff2) is not preloaded via a <link rel="preload"> tag. The browser only discovers it needs this font after parsing the CSS, adding unnecessary latency to font loading and delaying text rendering.
Recommendation: Add a preload hint in <head>:
<link rel="preload" href="/fonts/custom.woff2" as="font" type="font/woff2" crossorigin>
Single font format declared Low
Line 15
Only the woff2 format is provided. While woff2 has excellent modern browser support (~97%), older browsers may not render the custom font at all without a woff fallback.
Recommendation: Add a woff fallback for broader compatibility:
src: url("custom.woff2") format("woff2"), url("custom.woff") format("woff");

3. Core Web Vitals Issues

Render-blocking external CSS (LCP) High
Line 10
The synchronous <link rel="stylesheet" href="https://cdn.example.com/styles.css"> in <head> blocks all rendering until the entire CSS file downloads from an external CDN. This is one of the most impactful bottlenecks for LCP, especially on slow connections. The browser cannot paint anything until this resource is fetched and parsed.
Recommendation: Inline critical above-the-fold CSS directly in a <style> tag, then load the full stylesheet asynchronously:
<link rel="preload" href="styles.css" as="style" onload="this.rel='stylesheet'">
Consider self-hosting the CSS to reduce DNS and connection overhead.
No preloading for LCP image (LCP) Medium
Line 27
The hero image (/public/hero.png, 800x400) is likely the Largest Contentful Paint element, but it is not preloaded. The browser only starts downloading it after parsing the HTML to that point, missing an opportunity for earlier discovery.
Recommendation: Add a preload hint in <head>:
<link rel="preload" as="image" href="/public/hero.png">
LCP image uses unoptimized PNG format (LCP) Low
Line 27
The hero image uses PNG format. Modern image formats like WebP and AVIF offer 25-50% smaller file sizes with equivalent or better visual quality, which directly improves LCP by reducing download time.
Recommendation: Convert to modern formats and use the <picture> element with fallback:
<picture><source srcset="hero.avif" type="image/avif"><source srcset="hero.webp" type="image/webp"><img src="hero.png" ...></picture>
Image missing width and height attributes (CLS) High
Line 33
The promo.png image tag lacks explicit width and height attributes. Without these, the browser cannot reserve the correct space before the image loads, causing the page layout to shift when the image appears. This directly increases Cumulative Layout Shift (CLS).
Recommendation: Add explicit width and height attributes matching the image's intrinsic dimensions:
<img src="/public/promo.png" alt="Promotional banner" width="XXX" height="YYY">
Alternatively, use CSS aspect-ratio to reserve space.
Font swap layout shift potential (CLS) Medium
Lines 13-17
Even after adding font-display: swap, swapping from a system fallback font to the custom font can cause layout shift if the font metrics (character width, line height, ascent, descent) differ significantly between the two fonts.
Recommendation: Use CSS font metric overrides to match the fallback font to the custom font's metrics:
@font-face { font-family: "CustomFont-fallback"; src: local("Arial"); ascent-override: 90%; descent-override: 20%; line-gap-override: 0%; size-adjust: 105%; }
Synchronous inline JavaScript (INP) Medium
Lines 37-44
Synchronous JavaScript executing on the main thread can block user interaction responsiveness. While the current script is small, this pattern scales poorly. If the script triggers DOM reflows or grows in complexity, it will directly impact Interaction to Next Paint (INP).
Recommendation: Defer non-critical JavaScript execution. Use requestIdleCallback or setTimeout(..., 0) for non-urgent DOM updates. For critical interactions, keep handlers lean and avoid layout thrashing.
