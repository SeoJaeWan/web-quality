File: index.html | Review Date: 2026-03-11 | Focus Areas: JavaScript Loading, Font Optimization, Core Web Vitals

Summary: 7 issues identified across JavaScript loading, font optimization, and Core Web Vitals areas. 3 high severity, 3 medium severity, 1 low severity. The page has significant render-blocking resources, missing font optimization, and layout stability problems that negatively impact SEO performance and user experience.

1. JavaScript Loading Issues

[HIGH] Render-blocking inline script in <body>

Line: 37-44

Problem: An inline <script> block at the end of <body> executes synchronously. While it is placed at the bottom of the body (which is better than the head), it still blocks the parser at that point. The script uses document.getElementById and innerHTML, performing DOM manipulation synchronously. This contributes to longer Total Blocking Time (TBT) and can delay First Contentful Paint (FCP) if the browser hasn't finished rendering.

SEO Impact: Synchronous JavaScript execution increases TBT, one of the Core Web Vitals proxies. Google's page experience signals penalize pages with high TBT.

Recommendation:

  • Wrap the code in a DOMContentLoaded event listener or use requestIdleCallback.
  • Alternatively, move it to an external file and load with defer attribute: <script src="main.js" defer></script>.
<script defer src="/js/main.js"></script>

[MEDIUM] No async/defer scripts used anywhere

Line: N/A (absence)

Problem: The page contains no external JavaScript files loaded with async or defer attributes. All scripting is inline and synchronous. For pages that grow in complexity, this pattern does not scale and will cause increasing render-blocking time.

Recommendation: Extract JavaScript into external files and use defer for scripts that depend on DOM, or async for independent scripts (e.g., analytics).

[MEDIUM] innerHTML usage creates XSS risk and potential rendering issues

Line: 42

Problem: label.innerHTML = "Hello, " + userInput directly injects unsanitized user input from location.hash into the DOM. Beyond the security risk (XSS), this can cause unexpected re-renders and layout shifts if malicious or unexpected content is injected, impacting CLS.

Recommendation: Use textContent instead of innerHTML to prevent XSS and avoid unintended layout changes:

label.textContent = "Hello, " + userInput;

2. Font Optimization Issues

[HIGH] Missing font-display: swap in @font-face declaration

Line: 13-17

Problem: The @font-face rule for "CustomFont" does not include font-display: swap. Without this property, the browser uses the default behavior (typically font-display: block), which causes invisible text (FOIT - Flash of Invisible Text) for up to 3 seconds while the font loads. This directly harms:

  • FCP (First Contentful Paint): Text is invisible during font loading, delaying meaningful content display.
  • LCP (Largest Contentful Paint): If the custom font is used for the largest text element, LCP is delayed until the font loads.
  • CLS (Cumulative Layout Shift): When the font finally loads and swaps in, text dimensions may change, causing layout shift.

Recommendation: Add font-display: swap to show fallback text immediately:

@font-face {
  font-family: "CustomFont";
  src: url("/fonts/custom.woff2") format("woff2");
  font-display: swap;
}

[MEDIUM] No font preloading

Line: N/A (absence in <head>)

Problem: The custom font file /fonts/custom.woff2 is not preloaded. The browser only discovers the font when it parses the CSS, which delays font loading and consequently delays text rendering.

Recommendation: Add a <link rel="preload"> in the <head>:

<link rel="preload" href="/fonts/custom.woff2" as="font" type="font/woff2" crossorigin />

3. Core Web Vitals Issues

[HIGH] Image missing width/height attributes causes CLS

Line: 33

Problem: The promotional banner image (/public/promo.png) lacks width and height attributes. When the image loads, the browser must recalculate layout, causing a Cumulative Layout Shift (CLS). CLS is a Core Web Vital directly measured by Google for ranking.

Recommendation: Always specify width and height attributes on <img> elements to allow the browser to reserve space:

<img src="/public/promo.png" alt="Promotional banner" width="800" height="300" />

[HIGH (already noted)] Render-blocking external CSS

Line: 10

Problem: The external stylesheet https://cdn.example.com/styles.css is loaded synchronously in the <head> without any optimization. This blocks rendering until the CSS is fully downloaded and parsed, directly impacting FCP and LCP.

Recommendation:

  • Inline critical CSS directly in the <head> for above-the-fold content.
  • Load non-critical CSS asynchronously using the media/onload pattern:
<link rel="preload" href="https://cdn.example.com/styles.css" as="style" onload="this.onload=null;this.rel='stylesheet'" />
<noscript><link rel="stylesheet" href="https://cdn.example.com/styles.css" /></noscript>

[LOW] No resource hints for external domain

Line: N/A (absence)

Problem: The page loads resources from cdn.example.com but does not include dns-prefetch or preconnect hints. This adds DNS resolution and connection time before the resource can begin downloading.

Recommendation: Add resource hints early in the <head>:

<link rel="preconnect" href="https://cdn.example.com" />
<link rel="dns-prefetch" href="https://cdn.example.com" />
